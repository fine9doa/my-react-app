# 오뚜기 웹사이트 리액트 변환 가이드 (퍼블리셔용)
나는 리액트를 모르는 퍼블리셔야.
이 폴더의 모든 파일을 분석하여 리액트로 변환하되, 아래 지침을 엄격히 준수해줘.

## 1. 구조 설계 (Architecture)
- 모든 섹션은 독립적인 컴포넌트로 분리한다.
- 'App.jsx': 메인 본체
- 'Header.jsx', 'Footer.jsx': 공통 레이아웃
- 'Visual.jsx', 'Product.jsx', 'Brand.jsx', 'Sns.jsx' 등: main.html의 모든 섹션을 하나도 빠짐없이 쪼개서 생성할 것.

## 2. 작업 순서 (Workflow)
1. 환경 설정: 필요한 라이브러리(swiper, aos, gsap 등) 설치 명령어 안내 및 폴더 구조 생성.
2. 개별 코딩: 각 컴포넌트를 하나씩 작성하고 사용자(나)의 승인을 받은 후 다음 단계로 이동.
3. 통합: 모든 조각을 합쳐 완성본 제출.

## 3. 절대 규칙 (Critical Rules)
- 클래스명 보존: HTML의 'class'를 리액트의 'className'으로 바꿀 때 이름을 절대 수정하지 말기. (기존 CSS와의 호환성 유지)
- 에셋 경로 표준화:
    모든 정적 파일의 경로는 아래의 src/assets/ 하위 구조를 엄격히 따른다. (파일이 업로드되지 않았더라도 이 경로를 기준으로 코드를 작성할 것)
    (1) 이미지: src/assets/images/
    (2) 비디오: src/assets/video/
    (3) 폰트: src/assets/fonts/ (CSS @font-face 작성 시 반드시 적용)
- 한국어 주석: 로직이 복잡하거나 JS 기능을 옮긴 경우, "기존 common.js의 ~번 줄 내용임"이라는 설명을 포함하여 한국어로 상세히 주석을 달 것.
- 코드 중략 금지: 파일이 길어지더라도 절대로 코드를 생략(// ... 기존 코드 동일 등)하지 말고 전체 소스 코드를 모두 출력하라.

## 4. 커뮤니케이션
- 리액트 전문 용어보다는 퍼블리셔가 이해할 수 있는 쉬운 용어로 설명하라.
- 각 단계가 끝나면 반드시 "다음 단계를 진행할까요?"라고 물어볼 것.
